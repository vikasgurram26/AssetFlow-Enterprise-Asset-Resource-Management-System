PART 1: INTRO, ARCHITECTURE, DB SCHEMA & ROLE-BASED ACCESS CONTROL (Presenter 1)
--------------------------------------------------------------------------------
[Timing: 0:00 - 2:30 | Words: ~420]

[Visual Cue: Project Sign-in screen loaded in browser, displaying a sleek dark theme]

PRESENTER 1:
"Hello everyone, and welcome. Today, our team is proud to present AssetFlow: an 
Enterprise Asset & Resource Management System. Over the next ten minutes, we will 
guide you through its architecture, database design, and key operational lifecycles.

Let's start under the hood. AssetFlow is built as a single-app Django 6.0.7 project 
leveraging an SQLite database. Rather than relying on frontend logic to enforce rules, 
we designed our business constraints directly inside the Django database models. This 
approach ensures that whether data is altered via the custom UI, the Django Admin 
console, or database migrations, all state changes remain valid and consistent.

To ensure production security, we have hardened the application settings using environment 
variables loaded via a `.env` file. Critical settings like `DJANGO_SECRET_KEY`, 
`DJANGO_DEBUG`, and `DJANGO_ALLOWED_HOSTS` are kept out of source control. We have also 
enabled production-ready HTTP security headers to safeguard user data.

On the frontend, we use Tailwind CSS loaded via CDN, coupled with Material Symbols for 
clean iconography. 

A core pillar of the project is its granular, four-tier Role-Based Access Control (RBAC) 
framework, mapped to a custom User model extending Django's default AbstractUser:

1. Employee: The default role granted during sign-up. They can request transfers, 
   book bookable assets, file maintenance tickets, and view shared directory lists.
2. Department Head: Responsible for supervising a department. They can also approve 
   or reject transfer requests within their department.
3. Asset Manager: The operational controller. They can register, edit, allocate, 
   and return assets, manage maintenance, and run physical audits.
4. Administrator: The platform configuration head. They can manage departments and 
   categories, and promote employees to Department Head or Asset Manager.

Let me log in as an Admin to show you the initial setup."

[Action Cue: Presenter 1 types the admin credentials (username: 'admin', password: 'admin1234') and signs in]

PRESENTER 1:
"Upon logging in, the Django middleware identifies the role and renders our main dashboard. 
Let's navigate to the 'Organization' module.

Here, we manage the organizational hierarchy. Under the 'Departments' tab, we can establish 
departments, link them to parents, and appoint Department Heads. Under 'Asset Categories', 
we register categories and specify warranty periods. 

In the 'Employee Directory' tab, administrators promote newly registered employees to 
Department Heads or Asset Managers, and link them to their departments. 

Now, I will pass the microphone to [Presenter 2's Name] to show you how we register and manage 
assets through their operational lifecycle."

[Transition Cue: Presenter 1 passes presenter control to Presenter 2]

PRESENTER 2:
"Thank you, [Presenter 1's Name]. Let's explore the Asset Registry. 

Our asset database is centralized under the 'Assets' menu. The registry tracks a wealth of metadata 
including serial numbers, acquisition costs, condition states, and location names. 

Let's register a new asset."

[Action Cue: Presenter 2 clicks 'Register Asset' or navigation link. Fills in:

PRESENTER 2:
"When we save, the system runs model-level validations:
- Serial numbers must be unique across the entire database to prevent duplicate logging.
- The system automatically generates a unique Asset Tag in the sequential 'AF-XXXX' format.

Once registered, we can manage the asset's lifecycle. Assets transition through several 
statuses: Available, Allocated, Reserved, Under Maintenance, Lost, Retired, or Disposed.

Let's first look at our new Bulk Asset Retirement feature. In the Asset Directory list, 
we can select multiple assets using the checkboxes."

[Action Cue: Check the checkboxes next to two test assets in the list. See the 'Delete' button appear.]

PRESENTER 2:
"When rows are checked, a 'Delete' button dynamically appears next to the filters. 
If we click this button, the system processes a bulk update, marking the status of the selected 
assets as 'Retired'. This performs a 'soft delete' in the database, preserving all past allocation, 
maintenance, and audit history of those assets for future compliance tracking, rather than wiping out records.

Now, let's execute a Direct Allocation. Since our MacBook is 'Available', I can assign it 
directly to an employee or a department."

[Action Cue: On the MacBook's Detail page, under the Allocation section, select employee 'priya' 
 from the active list. Enter an expected return date 2 weeks in the future. Click 'Allocate Asset'.]

PRESENTER 2:
"The asset status changes to 'Allocated'. The recipient, Priya, receives an automatic in-app 
notification. If I attempt to allocate this MacBook to another user while it is already allocated, 
the database model blocks it, displays the current holder, and advises me to request a transfer.

When Priya returns the laptop, an Asset Manager enters condition notes and clicks 'Mark Returned', 
returning it to 'Available' status.

Alternatively, users can request a direct 'Transfer Request' to pass custody directly between 
employees. A Department Head or Asset Manager can review and approve it from the 'Allocation & Transfer' 
queue, which updates the custody records cleanly.

Now, [Presenter 3's Name] will walk you through the Resource Booking and Maintenance Ticket modules."

PART 3: SHARED BOOKING SYSTEM & KANBAN MAINTENANCE WORKFLOW (Presenter 3)
--------------------------------------------------------------------------------
[Timing: 5:00 - 7:30 | Words: ~410]

[Visual Cue: Resource Booking screen showing daily timeline bars and booking history]

PRESENTER 3:
"Thank you, [Presenter 2's Name].

Let's look at the 'Resource Booking' module. While allocations represent long-term custody, 
bookings are designed for short-term reservations of shared resources, like conference rooms or test devices.

To book an asset, the asset's 'Bookable' flag must be enabled. The system features a custom timeline 
view. Let's reserve our MacBook Pro for a time slot."

[Action Cue: Presenter 3 clicks 'Book Resource' or navigates to bookings/new/. Selects:
 - Asset: 'MacBook Pro M3 Max'
 - Start Date/Time: Select today at 10:00 AM
 - End Date/Time: Select today at 12:00 PM
 - Click 'Book']

PRESENTER 3:
"The booking is created successfully and rendered on our interactive daily timeline. 
But how does AssetFlow handle scheduling conflicts? The model includes overlap validation. 
Let's try to book the same MacBook Pro from 11:00 AM to 1:00 PM today."

[Action Cue: Try to submit a conflicting booking for the same MacBook from 11:00 AM to 1:00 PM today]

PRESENTER 3:
"The system blocks the submission and displays a clear error: 'End time must be after start time' or 
identifies the conflicting booking interval. Cancelled bookings are excluded from this validation.

Now, let's explore maintenance. Equipment breaks down. Rather than using emails, AssetFlow 
includes a complete Maintenance Lifecycle."

[Action Cue: Navigate to the 'Maintenance' tab, displaying the Kanban-style board]

PRESENTER 3:
"We present maintenance requests as a Kanban board with five columns: Pending, Approved, 
Tech Assigned, In Progress, and Resolved. 

Let's trace a ticket's journey:
1. Issue Reported: Any signed-in user can raise a ticket. It starts in 'Pending'. The asset's 
   status remains unchanged at this point.
2. Approval: An Asset Manager reviews the ticket and approves it. The asset's status automatically 
   transitions to 'Under Maintenance', blocking any allocations or bookings.
3. Assignment: The manager assigns a technician by name.
4. Work Starts: Once work begins, the ticket status changes to 'In Progress'. Note that the 
   progress bar on the card is a visual cue to indicate active repairs.
5. Resolution: When the repair is complete, the manager clicks 'Resolve Repair'. The ticket is 
   archived, the requester is notified, and the asset status transitions back to 'Available'.

Next, [Presenter 4's Name] will show you how we enforce physical verification through audits, 
and wrap up with analytics and future system plans."

[Transition Cue: Presenter 3 passes presenter control to Presenter 4]

PART 4: PHYSICAL AUDIT CYCLES, REPORTING & FUTURE SCOPE (Presenter 4)
--------------------------------------------------------------------------------
[Timing: 7:30 - 10:00 | Words: ~450]

[Visual Cue: Audit Cycle list screen showing open and closed audit cycles]

PRESENTER 4:
"Thank you, [Presenter 3's Name].

Let's discuss physical inventory tracking. Asset registers often drift from reality. To solve 
this, AssetFlow implements scoped 'Audit Cycles'. 

An Asset Manager can create an audit cycle, optionally scoping it by department or location. 
For example, we can create an audit scoped to 'Conference Room B'. The system automatically 
generates verification checklists for all assets located in that room."

[Action Cue: Navigate to Audit workspace, show list of items in the current audit cycle]

PRESENTER 4:
"Assigned auditors open the workspace and inspect each asset. For each item, they select:
- Verified: The asset is present and functional.
- Damaged: The asset is present but requires maintenance.
- Missing: The asset is unaccounted for.

If an asset is marked 'Missing' or 'Damaged', the system alerts all assigned auditors. 
Let's see what happens when the manager closes the audit cycle."

[Action Cue: Click 'Close Audit Cycle' and confirm the pop-up/prompt]

PRESENTER 4:
"Once closed, the audit becomes read-only. Crucially, the system executes an automated batch transition: 
every asset marked 'Missing' is transitioned to 'Lost'. This ensures that the register is updated 
and lost items are flagged.

To oversee these operations, we look to the Dashboard and Reports."

[Action Cue: Navigate to Dashboard, then Reports]

PRESENTER 4:
"The Dashboard provides real-time counts, overdue return flags, and recent activity logs. 
Under 'Reports & Analytics', we generate department breakdowns, category allocation ratios, 
and operational insights such as:
- Most Maintained Assets: Identifying hardware prone to failure.
- Idle Assets: Listing oldest available assets to optimize asset utilization.

Managers can also download the complete database register by clicking 'Export CSV'."

[Action Cue: Click 'Export CSV' to trigger the download of assetflow_report.csv]

PRESENTER 4:
"Before we close, we want to address the limitations of the current phase-1 release, which 
serves as a roadmap for phase 2:
1. Database: We currently use SQLite for development; we plan to transition to PostgreSQL 
   for production clustering.
2. QR/Barcode Scanning: The 'Scan Tag' button on the audit screen is a placeholder; phase 2 
   will integrate mobile camera scanning.
3. Communications: Notifications are currently in-app only; we plan to add SMTP mail integration 
   for automated overdue return alerts.
4. UI Polishing: The dashboard charts and timeline widgets use static segments; we plan to link 
   them to real-time charting libraries.

This concludes our demonstration of the AssetFlow system. Thank you for your time, and we'd be happy 
to answer any questions you have."
